How to get ready to start coding!


Author: SWH 3/1/2018

Developing an MVP (Minimum Viable Product)
  - where coding will be outsourced
  - delivery time is important
  - cost is critical
  - ability to make ongoing changes

Technologies (libraries, hosting, external webservices), mediums (realtime data, mobile, OG support, search), and how/where to find developers are the things this documents suggests to implement your product functionality.

Our Motto: More code costs more now and later!
Key Technique: Data Driven Design (add and remove fields using data)

Important things to always keep in mind...
  How much line count do we get? (avoid automated testing, use popular libraries)
  Who can we hire later to work on this? (try to do things the "normal" way)
  How long will simple or moderate changes take? (use good design and coding techniques)

Git Source Control Setup

Use GitHub to host your source code if you don't already have other ideas. Get the minimum (paid) subscription for private repositories. Source control is a ledger of changes and comments. It's a record of all work, good, bad or otherwise. You need to control this, but most of the action will happen from the developers here.

If you have specific UI needs...and you do!
such as your product centers on a displaying graphs, calenders, webgl graphics, etc, you might want to select the specific UI components first, then figure out which (if any) layout and mvc library will work best. This will also help determine the charteristics of the backend (such as notifications, OG tags, authorization, email, SMS, etc). Finding a developer with previous experience for your UI components can also help determine what choices libaries throughout the rest of your application.

If a UI component is popular (download/install count), that's a good sign it works now and will work later. Only use popular libraries.

To use a layout library?

All projects should focus on the mobile interface first. Most all popular libraries and frameworks allow you to do this.

Different devices require interfaces that respond to touch, portrait or landscape view, low bandwidth or dpi, etc.

Material Design from Google shows lots of promise, especially for Android applications that want to look uniform. Boostrap is older, has a much larger range of third party components, but still depends on jQuery and carries a lot of baggage from supporting older browsers.

I recommend using Bootstrap if possible - it's the most popular and probably will be for a while.

To use a MVC(binding) library?

Users always like fast applications. Transmitting and / or processing lots of data on demand is never fast.

By providing design guildlines and techniques, these libaries produce a much more uniform code base - multiple developers are able to work seperately and combine their work with a large amount of consistency.

They are template and binding libraries in practice and their use should be limited to that.

These libraries have lots of coding examples - great for less experienced programmers.

Defining your fields with data marginalizes the importance of HTML templates (the layout is data driven - not HTML). Screens will vary - so selecting and using a layout library is more key then the templates themselves.

Unless your application requires HTML templates with data binding - I recommend not using one.

Angular, React and Vue are popular.

When selecting the MVC library, choosing ES6 or Typescript comes into play.
  - Typescript has to be compiled.
  - Angular requires Typescript (whether they admit it or not).
  - React is lighter then Angular, and a bit more popular.
  - Vue is the newest and lightest - easy to read.

Applications should store state on the server, and display a thin layer on the client.

Use the minimum number of components needed, not the maximum. Each component can require it's own HTML, CSS and JS element (file).

Phone Application Features (Search, Notifications)

When deploying a javascript / web hosted application for iOS or Android, you'll need to wrap it with a native application. Electron is the most popular choice for new applications right now. Deploying to the store is no big deal, accessing device specific features can be a lot more work. If you require more then notifications, it's a good idea to isolate and control those features early in the project so they don't interfere with interface and database development.

Should I use jQuery?

No - unless... you use UI components that require it (i.e. Boostrap). ES6 handles DOM manipulation just fine. While there are a ton of popular components that require jQuery, there isn't any reason for new components being developed to depend on it.

Data Hosting

I prefer Firebase RTDB - especially for the immediate data that drives the interface. Also for storing any data that will be used for pushing notifications to mobile and other devices.

A heed of warning - NoSQL data design not the same as SQL schema design, and a developer that tells you there isn't much difference isn't experienced. This isn't something to reject a developer on, as if you can do SQL design, you'll be able to figure out the NoSQL design at somepoint, but NoSQL tends to carry a hierarchal tree of data related to item, for example a spreadsheet with all of it's sheets and cells stored under one place, another spreadsheet has all of it's data in another seperate place (document driven) - where SQL likes to break things up into their parts and join them back together, so all cells are stored in once place, all sheets are stored (without cells) in another place, and put together just when you need them. The document driven is far superior for supporting realtime/pushed data.

Authentication and Server Hosting

While there are a range of solutions, Firebase handles the authentication and application hosting with minimal setup and a great set of dev ops tools that install and are well known. AWS, S3 and Beanstalk are superior solutions for productions in many cases, for MVP development Firebase is hard to beat for how much time it takes setup and deploy applications. Heavy data processing needs should be isolated on EC2 or RDS and the user interface layer stored in Firebase RTDB.

Ask for Examples, Expect Fast Results, Be Willing To Pay For Time Spent

Most applications are not available for former developers to login and play with - at the same time demand to see some recent examples somewhere.

Don't let projects drag on - keep a short time from start to finish on a project.

Predicting how long a project will take requires knowing how it will be designed.

Expert people are unlikely to design and estimate the cost to implement your technology project for free.

Hire a developer with an existing build process and code base.

You want to spend your time and energy creating the features specific to your product - not figuring out the newest libraries and technologies.
